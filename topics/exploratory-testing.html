🧭 อะไรคือ Exploratory Testing?

"Exploratory testing is simultaneous learning, test design, and test execution." — Cem Kaner

Exploratory Testing คือการทดสอบที่ไม่มี Test Script ล่วงหน้า แต่ผู้ทดสอบจะ:

  • 🧠 เรียนรู้ ระบบไปพร้อมกับการทดสอบ
  • ✏️ ออกแบบ Test ใหม่ๆ ตามสิ่งที่ค้นพบ
  • ▶️ ทดสอบ ทันทีที่คิดได้

💡 ทำไมต้อง Exploratory Testing?

  • 🐛 พบ Bug ที่ Scripted Test หาไม่เจอ (20-30% ของ Serious Bugs)
  • ⚡ รวดเร็ว ไม่ต้องเตรียมเอกสารมาก
  • 🎨 ใช้ความคิดสร้างสรรค์และประสบการณ์ของ Tester
  • 🔍 เหมาะกับสถานการณ์ที่ไม่แน่นอน หรือระบบที่เปลี่ยนเร็ว

⚖️ Exploratory vs Scripted Testing

Exploratory Testing Scripted Testing
ไม่มี Test Script ล่วงหน้า มี Test Script/Case เขียนไว้แล้ว
Test Design + Execution พร้อมกัน Test Design ก่อน แล้วค่อย Execute
ใช้ความคิดสร้างสรรค์ ทำตาม Steps ที่กำหนด
พบ Bug แปลกๆ ที่ไม่คาดคิด ยืนยันว่า Features ทำงานตาม Spec
ต้องใช้ Tester ที่มีประสบการณ์ Junior Tester ก็ทำได้
ไม่ Repeatable 100% Repeatable เหมือนกันทุกครั้ง

🤝 ใช้ร่วมกัน!

การทดสอบที่ดีควรมีทั้งสองแบบ: Scripted Test สำหรับ Regression และ Requirements หลัก + Exploratory Test สำหรับค้นหา Bug ใหม่ๆ

⏱️ Session-Based Test Management (SBTM)

วิธีจัดการ Exploratory Testing ให้มีโครงสร้าง โดยแบ่งเป็น "Session" (ช่วงเวลาทดสอบ)

🕐 Session Types

⏰

Short Session

60-90 นาที

สำหรับ: ฟีเจอร์เล็ก หรือตรวจสอบเร็ว
⏱️

Normal Session

90-120 นาที

สำหรับ: ฟีเจอร์มาตรฐาน
🕐

Long Session

2-3 ชั่วโมง

สำหรับ: ฟีเจอร์ซับซ้อน หรือ End-to-End

💡 ตัวอย่าง Charters

Charter 1: Shopping Cart

สำรวจ ฟีเจอร์ Shopping Cart และ Checkout Flow โดยใช้ บัญชีทดสอบหลายประเภท (มีสินค้า/ไม่มีสินค้า/สินค้าหมดสต็อก) เพื่อ หาปัญหาการคำนวณราคา หรือข้อผิดพลาดในการ Checkout

Charter 2: Search Function

ตรวจสอบ การทำงานของระบบค้นหา โดยใช้ คำค้นหาพิเศษ (SQL injection patterns, XSS, long strings) เพื่อ ประเมินความปลอดภัยและความทนทานของระบบ
