🔄 TDD (Test Driven Development)

TDD คือแนวทางการพัฒนาซอฟต์แวร์ที่เขียน Test ก่อน แล้วค่อยเขียน Code ให้ผ่าน Test นั้น

💡 TDD คืออะไร?

TDD ไม่ใช่การทดสอบแบบดั้งเดิม แต่เป็น วิธีคิด (mindset) ในการออกแบบซอฟต์แวร์ผ่านการเขียน tests

🎯 ข้อดีของ TDD

  • ✅ รู้ว่า code ทำงานถูกต้องตั้งแต่เริ่ม
  • ✅ Design ดีขึ้น (loosely coupled, testable)
  • ✅ มี test coverage สูงโดยอัตโนมัติ
  • ✅ กล้าแก้ไข/refactor code เพราะมี test คุ้มครอง
  • ✅ Documentation ที่อัปเดตเสมอ (tests บอกว่า code ทำอะไร)

⚠️ ข้อควรระวัง

  • ❌ ใช้เวลาเรียนรู้มากขึ้นตอนเริ่ม
  • ❌ อาจเขียน test มากเกินจำเป็น (over-testing)
  • ❌ ไม่เหมาะกับ UI หรือ requirement ที่เปลี่ยนบ่อย

🚦 TDD Cycle: Red → Green → Refactor

    ┌─────────────────────────────────────┐
    │                                     │
    ▼                                     │
┌─────────┐    ┌──────────┐    ┌─────────┴──┐
│   RED   │───▶│  GREEN   │───▶│  REFACTOR  │
│ (Write  │    │(Write    │    │(Improve    │
│  Test)  │    │  Code)   │    │   Code)    │
└─────────┘    └──────────┘    └────────────┘
   (Fail)         (Pass)          (Pass)

🔴 1. RED - เขียน Test ที่ Fail

# test_calculator.py

def test_add():
    calc = Calculator()
    result = calc.add(2, 3)
    assert result == 5  # Test นี้จะ Fail เพราะยังไม่มี class Calculator

🟢 2. GREEN - เขียน Code ให้ Test ผ่าน

# calculator.py

class Calculator:
    def add(self, a, b):
        return a + b  # เขียนแค่พอให้ test ผ่าน

🔵 3. REFACTOR - ปรับปรุง Code

# ปรับปรุงโค้ดให้ดีขึ้น แต่ยังคงให้ test ผ่านเหมือนเดิม
# เช่น: เพิ่ม docstring, ปรับชื่อตัวแปร, ลดความซ้ำซ้อน

🎭 BDD (Behavior Driven Development)

BDD คือการขยายแนวคิด TDD ให้เข้าใจง่ายขึ้นโดยใช้ภาษาธรรมชาติ (Given-When-Then) เพื่อให้ทีม business, QA, และ dev เข้าใจตรงกัน

💡 BDD ต่างจาก TDD อย่างไร?

TDD: Test technical behavior (function นี้ return ค่าอะไร)
BDD: Test business behavior (user ทำอะไร ระบบตอบสนองอย่างไร)

🎯 ข้อดีของ BDD

  • ✅ ทีมเข้าใจ requirement ตรงกัน (ใช้ภาษาธรรมชาติ)
  • ✅ Living documentation (feature files อธิบายระบบ)
  • ✅ ลด communication gap ระหว่าง business และ tech

📝 Gherkin Syntax (Given-When-Then)

Gherkin เป็นภาษาที่ใช้เขียน BDD scenarios แบบ domain-specific

Keywords หลัก

Keywordความหมายตัวอย่าง
Featureชื่อฟีเจอร์Feature: Login
ScenarioTest case หนึ่งๆScenario: Successful login
GivenPreconditionGiven user is on login page
WhenActionWhen user enters valid credentials
ThenExpected resultThen user should see dashboard
And/Butเชื่อมต่อ stepAnd message "Welcome" displayed

💡 Example: Login Feature

Feature: User Login
  As a registered user
  I want to login with my credentials
  So that I can access my account

  Scenario: Successful login with valid credentials
    Given user is on the login page
    And user has a valid account
    When user enters "john@example.com" in email field
    And user enters "password123" in password field
    And user clicks the login button
    Then user should be redirected to dashboard
    And welcome message should be displayed

  Scenario: Failed login with invalid password
    Given user is on the login page
    When user enters valid email "john@example.com"
    And user enters invalid password "wrongpass"
    And user clicks the login button
    Then error message "Invalid credentials" should be displayed
    And user should remain on login page

🔧 Tools สำหรับ BDD

  • Cucumber - Java, JavaScript, Ruby, Python (Behave)
  • SpecFlow - .NET
  • Behave - Python
  • pytest-bdd - Python
  • Robot Framework - Keyword-driven testing

⚖️ TDD vs BDD Comparison

TDD BDD
Test technical units Test business behavior
เขียนโดย developers เขียนโดยทั้ง team (3 Amigos)
ใช้ xUnit frameworks ใช้ Gherkin + step definitions
Focus: Code correctness Focus: Business value
Tests = Technical docs Tests = Living docs for all

💡 ใช้ร่วมกันได้!

TDD และ BDD ไม่ใช่การเลือกข้าง หลายทีมใช้ BDD สำหรับ high-level acceptance tests และ TDD สำหรับ unit tests ภายใน
